# 변경 관리 가이드

## 1. 개요

이 문서는 프로젝트의 변경 사항을 체계적으로 관리하기 위한 가이드입니다.

## 2. 변경 관리 체계

```
┌─────────────────────────────────────────────────────────────┐
│                    변경 관리 문서 체계                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [CHANGELOG.md]          버전별 변경 이력                    │
│       │                  - 기능 추가/변경/삭제               │
│       │                  - 버그 수정                        │
│       │                  - 릴리스 노트                      │
│       │                                                     │
│  [doc/adr/]              아키텍처 결정 기록                  │
│       │                  - 기술 선택 이유                    │
│       │                  - 설계 결정 배경                    │
│       │                  - 대안 검토 내용                    │
│       │                                                     │
│  [doc/WBS.md]            작업 진행 상황                      │
│       │                  - 전체 작업 목록                    │
│       │                  - 완료/진행/대기 상태               │
│       │                                                     │
│  [Git Commits]           세부 변경 이력                      │
│                          - Conventional Commits             │
│                          - 코드 레벨 변경                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 문서별 용도

### 3.1 CHANGELOG.md

**용도:** 사용자/이해관계자를 위한 버전별 변경 이력

**작성 시점:**
- 새 버전 릴리스 시
- 주요 기능 완료 시

**포함 내용:**
- 새로운 기능 (Added)
- 변경된 기능 (Changed)
- 버그 수정 (Fixed)
- 보안 업데이트 (Security)

**예시:**
```markdown
## [0.3.0] - 2025-02-15

### Added
- 사용자 답안 입력 기능
- 즉시 채점 기능
- 순위 확인 기능
```

### 3.2 ADR (Architecture Decision Records)

**용도:** 중요한 아키텍처/기술 결정 기록

**작성 시점:**
- 새로운 기술 스택 선택 시
- 아키텍처 패턴 결정 시
- 설계 방향 변경 시

**포함 내용:**
- 결정의 컨텍스트
- 검토한 대안들
- 최종 결정과 이유
- 예상되는 결과

**파일 위치:** `doc/adr/NNN-title.md`

**예시:**
```
doc/adr/
├── 000-template.md
├── 001-system-architecture.md
├── 002-batch-system-design.md
└── 003-chart-library-selection.md
```

### 3.3 WBS.md

**용도:** 전체 작업 목록 및 진행 상황 추적

**갱신 시점:**
- 새로운 작업 추가 시
- 작업 상태 변경 시
- 요구사항 변경 시

**포함 내용:**
- Phase별 작업 목록
- 작업 상태 (완료/진행/대기)
- 의존 관계
- 진행률

### 3.4 Git Commits

**용도:** 코드 레벨의 세부 변경 이력

**커밋 메시지 형식:**
```
<type>(<scope>): <subject>

<body>

<footer>
```

## 4. 변경 관리 프로세스

### 4.1 기능 개발 시

```
1. WBS에서 작업 상태를 "진행중"으로 변경
           ↓
2. 기능 개발 및 커밋 (Conventional Commits)
           ↓
3. WBS에서 작업 상태를 "완료"로 변경
           ↓
4. CHANGELOG의 [Unreleased] 섹션에 추가
           ↓
5. (릴리스 시) CHANGELOG 버전 태그 및 일자 추가
```

### 4.2 아키텍처 결정 시

```
1. ADR 템플릿 복사
           ↓
2. 컨텍스트, 대안, 결정 작성
           ↓
3. 팀 리뷰 (필요 시)
           ↓
4. 상태를 "Accepted"로 변경
           ↓
5. doc/adr/README.md 목록에 추가
           ↓
6. 커밋: "docs(adr): ADR-NNN 제목 추가"
```

### 4.3 릴리스 시

```
1. CHANGELOG [Unreleased] 내용을 새 버전으로 이동
           ↓
2. 버전 번호와 일자 추가
           ↓
3. Git 태그 생성: git tag v0.3.0
           ↓
4. GitHub Release 생성 (선택)
```

## 5. 자주 참조하는 상황별 문서

| 상황 | 참조 문서 |
|------|----------|
| "이 기능이 왜 이렇게 구현됐지?" | ADR |
| "지난 버전에서 뭐가 바뀌었지?" | CHANGELOG |
| "어떤 작업이 남아있지?" | WBS |
| "이 코드 언제 추가됐지?" | Git Log |
| "개발 규칙이 뭐였지?" | DEVELOPMENT_STANDARDS.md |

## 6. 문서 위치 요약

```
hopenvision/
├── CHANGELOG.md              # 버전별 변경 이력
├── doc/
│   ├── WBS.md                # 작업 목록 및 진행 상황
│   ├── PROJECT_OVERVIEW.md   # 프로젝트 개요
│   ├── IMPLEMENTATION_PROPOSAL.md  # 구현 제안서
│   ├── adr/                  # 아키텍처 결정 기록
│   │   ├── README.md
│   │   ├── 000-template.md
│   │   ├── 001-*.md
│   │   └── ...
│   └── dev/
│       ├── DEVELOPMENT_STANDARDS.md  # 개발 표준
│       ├── SERVER_CONFIGURATION.md   # 서버 설정
│       └── CHANGE_MANAGEMENT.md      # 이 문서
└── .git/                     # Git 이력
```

## 7. 권장 사항

1. **작은 단위로 자주 커밋**: 변경 추적 용이
2. **ADR은 결정 전에 작성**: 대안 검토 기록
3. **CHANGELOG는 사용자 관점**: 기술적 세부사항 최소화
4. **WBS는 실시간 갱신**: 진행 상황 투명성
